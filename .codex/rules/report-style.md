# 报告与文档输出规则

本规则适用于输出 Markdown 报告、分析文档、流程说明、架构说明、对外流转结论、阶段总结等场景。

## 适用优先级

- 用户明确要求的报告口径优先。
- 调试即时分析可按 `android-debug.md` 区分事实、推测、建议；但当用户要求生成正式报告或文档时，按本规则收敛表述。
- 代码流程梳理、架构说明、需求/方案设计不要写成 bug 分析报告。

## 正式报告口径

正式报告应基于事实数据直接陈述，避免把内部分析过程写进正文：

- 不写“置信度高/中/低”这类判断标签，除非用户明确要求。
- 不写“上一版、旧文件、补跑前、更新后变化”这类历史修订痕迹；直接按当前最新事实分析。
- 不写“我认为、可能大概、应该是”作为主结论开头；不确定内容放到“待确认”或“需要补充证据”。
- 不把工具执行过程、读取失败过程、权限绕路过程写进正式正文，除非这些是问题本身的证据。
- 数据口径要明确，例如结果目录、session、测试时间、模块名、去重键、排除条件。

## 推荐结构

技术分析报告优先使用：

1. 摘要结论：一句话说明当前结论、影响范围和优先级。
2. 数据来源：列出使用的日志、结果文件、源码分支、commit 窗口。
3. 关键统计：只放核心数字，避免长篇明细淹没结论。
4. 问题分组：按模块、症状、失败原因或调用链分组。
5. 证据说明：给出能支撑结论的文件、日志片段、源码路径或 commit。
6. 建议动作：给出最小复测、补充日志、代码检查或归属模块。
7. 附录：长列表、逐用例明细、原始命中结果放附录。

流程/架构文档优先使用：

1. 主链路总览。
2. 分层流程：应用、Framework、native/HAL、kernel、硬件或平台层。
3. 关键函数/接口。
4. 时序图或流程图。
5. 代码索引。
6. 注意事项或边界。

## 表述要求

- 结论先行，后放证据。
- 用事实数据说话，避免形容词堆叠。
- 如果同一文档用于评审或转交，保留“当前结论”和“下一步动作”，删掉聊天式解释。
- 如果用户要求精简，优先删长明细、重复证据、历史版本说明，不删核心统计和归因结论。

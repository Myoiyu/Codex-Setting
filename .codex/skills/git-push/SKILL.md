# Git Push Skill

## 功能概述

自动分析代码改动，生成规范格式的 commit message，并将修改推送到远程 Git 仓库的当前分支。

## 执行步骤

### Step 1：收集必要信息

在执行提交前，**必须**向用户询问以下信息：

- **问题单号**：请用户提供问题单号（如 DTS2024XXXXXX），若无则填"无"

使用 `ask_user_question` 工具向用户提问。

### Step 2：获取代码修改状态

运行以下命令查看改动：

```bash
git -C <workspace_path> diff --stat HEAD
git -C <workspace_path> status
git -C <workspace_path> diff HEAD
```

### Step 3：生成 commit message

根据代码修改内容和对话上下文，按照以下规则填写各字段：

#### 字段填写规则

| 字段 | 填写规则 |
|------|---------|
| **解决问题** | 根据代码修改内容和对话内容总结，描述本次修改解决了什么问题或实现了什么功能 |
| **提交分类** | 从以下选项选择最合适的一个：`配置修改` / `问题解决` / `模块开发` / `无` |
| **测试注意** | 填写 `系统` |
| **测试结果** | 填写 `自测通过` |
| **问题单号** | 用户提供的问题单号，若用户说"无"则填 `无` |
| **操作类型** | 从以下选项选择最合适的一个：`修改` / `新增` / `删除` |
| **重要程度** | 从以下选项选择最合适的一个：`重要` / `一般` / `无` |
| **影响产品** | 填写 `9655` |

#### 提交分类与问题单号的联动规则（重要）

> **如果问题单号为"无"或用户未提供问题单号，提交分类绝对不能选择"问题解决"**，应改选 `模块开发`、`配置修改` 或 `无`。
>
> **只有当问题单号有具体编号时**，才可以选择"问题解决"。

### Step 4：格式化 commit message

commit message 必须严格按照以下格式，每行一个字段，不能缺少任何字段：

```
【解决问题】：<内容>
【提交分类】：<内容>
【测试注意】：<内容>
【测试结果】：<内容>
【问题单号】：<内容>
【操作类型】：<内容>
【重要程度】：<内容>
【影响产品】：<内容>
```

### Step 5：执行 git 操作

依次执行以下命令（使用 PowerShell，分号分隔，不使用 &&）：

**注意**：commit message 中包含中文和换行，推荐写入临时文件后使用 `git commit -F` 方式提交：

```powershell
$msg = @"
【解决问题】：<内容>
【提交分类】：<内容>
【测试注意】：<内容>
【测试结果】：<内容>
【问题单号】：<内容>
【操作类型】：<内容>
【重要程度】：<内容>
【影响产品】：<内容>
"@
$msg | Out-File -FilePath "$env:TEMP\commit_msg.txt" -Encoding utf8
git -C <workspace_path> add .
git -C <workspace_path> commit -F "$env:TEMP\commit_msg.txt"
git -C <workspace_path> push origin HEAD
```

### Step 6：确认结果

检查命令输出，向用户报告：
- commit hash
- 推送到的远程分支
- 推送是否成功

若推送失败，报告错误信息并提示用户可能的原因。

```
【测试注意】：系统
【测试结果】：自测通过
【问题单号】：DTS2024001234
【操作类型】：修改

```
